# Chapter 2: The World State

The part of the Ethereum state that people most commonly associate with "state" is called the World State.

Most people really just call it the state and refer to the history as a distinct component, but as we saw earlier, history is part of the state if we were to view Ethereum as a state machine, since history can affect the state transition.

However, to be consistent with common terminology used in the space, we will simply call the World State the state from now on, and refer to history distinctly.  Just keep in mind that in fact, the history is part of the state and the state is World State + History.

## The three forms of state

One tricky thing to understand about the Ethereum state is that it can be represented in three forms, and it's a common source of confusion in Ethereum discourse when people are talking about state but they are actually talking about different forms.

The three forms are:

- The logical form
- The operational form
- The commitment form

We will focus on the logical and the commitment form, since the operational form differs by client implementation.

## State in logical form

Logically, the state of Ethereum can be described very simply as a two-tier data structure as described by the following TypeScript code:

```typescript
type Account {
    nonce: UInt;
    balance: UInt256;
    code: Bytes;
    storage: Map<Bytes32, Bytes32>;
}

type State = Map<Address, Account>;
```

## State in commitment form

To appreciate why Ethereum state uses the commitment scheme it does, let's try to design our own commitment scheme.  For example, why don't we just lay out all the data serially in a flat byte array, and then just hash that array?

While we may get determinism that way, we'd be missing out on:

- Very expensive to incrementally update the state
- Impossible to prove a piece of the state without serving the entire state


## Binary State Tree (EIP-7864)

Ethereum is moving toward a new binary tree structure to replace the current hexary Merkle Patricia Trie. The full proposal is described in [EIP-7864](https://eips.ethereum.org/EIPS/eip-7864).

The core idea: keys are 32 bytes, where the first 31 bytes form the *stem* and the last byte is the *subindex*. All 256 values sharing the same stem live in one `StemNode`. The path through the tree is determined bit-by-bit (MSB first) by the stem, with `InternalNode`s branching left (0) or right (1) at each level.

When two stems need to coexist, one `InternalNode` is created for **each bit of their shared prefix**, plus one final `InternalNode` where they diverge. This is in contrast to the MPT, which uses extension nodes to skip over shared-prefix bits in a single hop. The binary tree avoids extension nodes entirely — every level is a simple two-way branch — which makes it simpler to implement in ZK circuits.

## Tree Embedding (EIP-7864)

Rather than storing account data across separate trees, EIP-7864 co-locates everything for an account on a single **stem**. Each of the 256 subindex slots on that stem is assigned a specific role:

- **0x00** — `BASIC_DATA`: version, code size, nonce, and balance packed into one 32-byte leaf
- **0x01** — `CODE_HASH`: keccak256 of the account's bytecode
- **0x02 – 0x3F** — Reserved for future protocol use
- **0x40 – 0x7F** — First 64 storage slots (slots 0–63)
- **0x80 – 0xFF** — First 128 code chunks (31 bytes of code each)

When a contract has more than 64 storage slots or more than 128 code chunks, the extras spill to **overflow stems** derived from the same address.

## Hash Commitment (Merkleization)

Every time Ethereum's state changes, a compact fingerprint of the entire tree must be included in the block header. That fingerprint — the **state root** — is computed by *merkleizing* the binary tree bottom-up, following four rules:

- **Leaf**: `leaf_hash = hash(value)` — each slot value is hashed on its own.
- **Stem**: the 256 leaf hashes are first reduced into a binary subtree, then the stem bytes are mixed into an outer hash: `stem_hash = hash(stem ║ 0x00 ║ inner)`.
- **Internal**: `node_hash = hash(left_hash ║ right_hash)` — hashes bubble up from leaves to root.
- **Empty**: unset slots and absent branches contribute `0x00 × 32` — no hash call needed.

Changing a single leaf only requires recomputing `O(log n)` hashes along the path from that leaf to the root.
